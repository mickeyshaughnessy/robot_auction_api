Robot Services Exchange (RSX) FAQ

General Information

What is the Robot Services Exchange (RSX)?
The Robot Services Exchange (RSX) is a platform designed to facilitate the buying and selling of robot services. It acts as an intermediary between robot owners (sellers) and people who need robot services (buyers). Think of it as a job board for robots, where tasks are posted and robots are assigned to complete them.
How does RSX work?
RSX works through a bidding system:
  1. Buyers submit bids for specific services they need, including details like the type of service, location, price, and time frame.
  2. Robot owners (sellers) can browse available jobs and accept those that match their robots' capabilities and location.
  3. When a match is made, the buyer pays for the service, the robot performs the task, and then the seller is paid.
This system allows for dynamic pricing and efficient allocation of robot resources based on supply and demand.

For Buyers

How do I place a bid for a robot service?
To place a bid, you need to use the `/make_bid` endpoint of our API. Here's a step-by-step process:
  1. Ensure you have registered an account and obtained an authentication token.
  2. Prepare your bid data, including:
    • service: The type of service you need (e.g., cleaning, delivery)
    • lat: Latitude of the service location
    • lon: Longitude of the service location
    • price: Your bid price for the service (in USD)
    • end_time: The deadline for the service (in Unix timestamp format)
  3. Send a POST request to `https://100.26.236.1:5001/make_bid` with your bid data in JSON format, including your authentication token in the header.
The system will then process your bid and make it available for robot owners to accept.

RSX Protocol Details

How does a buyer client interact with the RSX protocol?
A buyer client interacts with the RSX protocol through the following steps:
  1. Authentication:
    • Register using POST request to `/register` with username and password.
    • Login using POST request to `/login` to receive an authentication token.
  2. Placing a Bid:
    • Send a POST request to `/make_bid` with the following JSON data: 
      {
    "service": "cleaning",
    "lat": 40.7128,
    "lon": -74.0060,
    "price": 50.00,
    "end_time": 1632528000
}
    • Include the authentication token in the request header.
  3. Checking Bid Status:
    • Use the `/nearby` endpoint to view recent activities, including bid status.
  4. Job Completion:
    • Once the job is done, use the `/sign_job` endpoint to rate the service and complete the transaction.
How does a seller (robot owner) client interact with the RSX protocol?
A seller client interacts with the RSX protocol as follows:
  1. Authentication:
    • Register and login similar to the buyer client to obtain an authentication token.
  2. Robot Registration:
    • Use the RobotOwnerClient to register robots with the following information: 
      client.register_robot(
    "robot1", 
    ["cleaning", "delivery"], 
    40.7128, 
    -74.0060, 
    10
)
  3. Job Searching:
    • Periodically send POST requests to the `/grab_job` endpoint with the robot's capabilities and location: 
      {
    "services": ["cleaning", "delivery"],
    "lat": 40.7128,
    "lon": -74.0060,
    "max_distance": 10
}
  4. Job Execution:
    • When a job is received, the client handles the job execution logic.
  5. Job Completion:
    • After completing the job, use the `/sign_job` endpoint to update the job status.
What are the key differences between buyer and seller interactions in the RSX protocol?
The key differences are:
  • Bid vs. Job: Buyers create bids for services, while sellers search for and accept jobs.
  • Proactive vs. Reactive: Buyers proactively place bids, while sellers reactively search for available jobs.
  • Data Provided: Buyers specify service details and price, while sellers provide robot capabilities and location.
  • Endpoints Used: Buyers primarily use `/make_bid` and `/nearby`, while sellers mainly use `/grab_job`.
  • Client Implementation: Buyer clients are typically user-facing applications, while seller clients (like RobotOwnerClient) often run autonomously.
How does the RSX protocol handle matching bids to robots?
The RSX protocol matches bids to robots through the following process:
  1. Bid Storage: When a buyer submits a bid, it's stored in the system with its details (service, location, price, etc.).
  2. Job Search: When a seller's robot searches for jobs, the system:
    • Filters bids based on the robot's capabilities (services it can perform).
    • Checks if the bid's location is within the robot's specified max_distance.
    • Considers the bid's end_time to ensure the robot can complete the job in time.
  3. Price Matching: If multiple robots qualify for a job, the one accepting the lowest price gets the job.
  4. Job Assignment: Once a match is found, the job is assigned to the robot and removed from the available bids.
This process ensures efficient matching of services to capable robots while respecting location constraints and competitive pricing.
How does the RSX protocol ensure fair competition among sellers?
The RSX protocol promotes fair competition through several mechanisms:
  1. Price-based Assignment: Jobs are typically assigned to the robot accepting the lowest price, encouraging competitive pricing.
  2. Location-based Matching: By considering robot locations and max_distance, the system ensures that distant robots don't unfairly compete with local ones.
  3. Capability Filtering: Only robots with the required capabilities are considered for each job, ensuring fair competition among qualified robots.
  4. Rating System: The `/sign_job` endpoint allows buyers to rate services, which can influence future job assignments, encouraging high-quality service.
  5. Equal Access to Jobs: The `/grab_job` endpoint provides all registered robots equal opportunity to search for and accept available jobs.

Technical Information

What technologies does RSX use?
RSX is built using a variety of modern technologies:
  • Backend: The core API is built with Python using the Flask framework.
  • Database: We use Redis for fast, in-memory data storage and retrieval.
  • Server: The application is served using Gunicorn, a Python WSGI HTTP Server.
  • Frontend: The dashboard uses HTML, CSS, and JavaScript with the Leaflet library for mapping.
  • API Communication: RESTful API endpoints are used for all client-server communication.
  • Authentication: We use JWT (JSON Web Tokens) for secure user authentication.
This tech stack allows for a scalable, efficient, and secure platform for robot service exchange.
